## 核心操作规程（所有任务强制执行）

> 这 9 条是起点中的起点，任何任务开始前必须已满足。

1. **强制任务拆解**：多步骤任务开始前必须生成 TodoList，每个 item 为原子操作（"改一行配置" / "跑一条命令" / "写一个测试"），禁止出现"完成 X 模块"这种模糊 item。
2. **强制命令超时**：任何 shell 命令必须带 `--timeout` 或 `timeout` 参数，禁止无期限运行。
3. **强制验收用例**：每个需求必须先定义验收用例（可自动化的 shell 命令或测试脚本），再写代码；代码是验收用例的实现，不是反过来。
4. **强制中文输出**：所有生成的文档、注释、输出信息均必须使用中文，禁止输出英文内容。
5. **强制 Linter**：代码提交前必须通过 linter 检查。
   - TypeScript / TSX 代码：用 **Biome**（`pnpm biome check ./src`）
   - Python 代码：用 **ruff**（`ruff check .`）
   - 两者均要求零错误（warnings 允许，errors 必须清零）
6. **禁止自动推送**：所有 `git push` 必须等待用户显式命令再执行，不得在自动流程中推送。
7. **反复问题聚焦原则**：当遇到反复改都改不好的问题时，必须：
   - 严格聚焦当前问题，避免分散精力到其他方向
   - 准确找到根因后再修复，不要猜测
   - 使用 `/autoresearch` 技能进行系统性迭代
   - 可开 agents team 进行多角度对抗性分析
   - 记录每次尝试和结果，直到彻底解决
8. **强制最小可行方案优先（MVP-First）**：任何需求实施必须严格遵循以下流程：
   - **第一步：识别 MVP**：从需求中提炼出最小可行方案——即能独立交付价值、可独立验证的最小功能子集。严禁一次性铺开所有功能。
   - **第二步：MVP 必须包含详尽验收用例**：每个 MVP 必须附带完整的验收用例清单（可执行的测试命令、预期输入/输出、边界条件），验收用例必须在编码前定义完毕。
   - **第三步：MVP 必须包含原子化 TodoList**：将 MVP 拆解为原子操作级别的 TodoList（与规则 1 一致），每个 item 可在 1-2 分钟内完成，标注前置依赖关系。
   - **第四步：非必要步骤列为可选项**：MVP 之外的功能、优化、重构等统一归入「可选增强项」清单，标注优先级（P1/P2/P3），仅在 MVP 验收通过且用户明确要求后才可实施。
   - **禁止行为**：
     - 禁止在 MVP 未验收通过前实施任何可选增强项
     - 禁止将「锦上添花」功能混入 MVP 范围
     - 禁止跳过验收用例定义直接编码
9. **强制基于事实回答（禁止幻觉）**：所有回答必须基于实际文件内容和可验证的事实，严禁凭空捏造：
   - **先读后答**：回答任何关于代码、配置、文件结构的问题时，必须先用工具读取相关文件确认实际内容，禁止凭记忆或推测作答。
   - **引用出处**：回答中涉及的代码片段、配置项、文件路径等必须标注来源文件及行号，确保可追溯。
   - **不确定即声明**：当无法通过工具确认事实时，必须明确告知用户「我无法确认此信息，需要进一步查看」，严禁编造看似合理的答案。
   - **禁止行为**：
     - 禁止编造不存在的文件路径、函数名、API、配置项
     - 禁止杜撰文件内容或代码逻辑
     - 禁止在未读取文件的情况下描述文件内容
     - 禁止用「应该是」「大概是」等模糊措辞掩盖不确定性

## 核心操作规程（所有任务强制执行）

> 这 11 条是起点中的起点，任何任务开始前必须已满足。

1. **强制任务拆解**：多步骤任务开始前必须生成 TodoList，每个 item 为原子操作（"改一行配置" / "跑一条命令" / "写一个测试"），禁止出现"完成 X 模块"这种模糊 item。
2. **强制命令超时**：任何 shell 命令必须带 `--timeout` 或 `timeout` 参数，禁止无期限运行。
3. **强制验收用例**：每个需求必须先定义验收用例（可自动化的 shell 命令或测试脚本），再写代码；代码是验收用例的实现，不是反过来。
4. **强制中文输出**：所有生成的文档、注释、输出信息均必须使用中文，禁止输出英文内容。
5. **强制 Linter**：代码提交前必须通过 linter 检查。
   - TypeScript / TSX 代码：用 **Biome**（`pnpm biome check ./src`）
   - Python 代码：用 **ruff**（`ruff check .`）
   - 两者均要求零错误（warnings 允许，errors 必须清零）
6. **强制 pre-commit review**：每次 `git commit` 前必须通过自动化 code review。
   - 触发方式：`.git/hooks/pre-commit` 已在仓库安装，commit 时自动执行
   - 审查脚本：`scripts/pre-commit-review.sh`
   - 审查维度：正确性 / 安全性 / 性能 / 可维护性 / 项目一致性（5个维度，详见 `.cursor/skills/code-review/SKILL.md`）
   - **必须修复的问题（阻塞提交）**：硬编码凭证/Token/API Key、危险函数（`eval()` / `new Function()` / `innerHTML` / `document.write()`）、空 `catch {}`、SQL 注入风险
   - 建议修改 / 可以更好：不阻塞提交，但会在 hook 输出中显示
   - **跳过审查**（仅限紧急 hotfix）：`SKIP_REVIEW=1 git commit -m "..."`
   - 适用豁免：初始空 commit / revert commit / 没有 staged 变更时自动跳过
7. **禁止自动推送**：所有 `git push` 必须等待用户显式命令再执行，不得在自动流程中推送。
8. **反复问题聚焦原则**：当遇到反复改都改不好的问题时，必须严格聚焦当前问题，准确找到根因后再修复，不要猜测。可使用 `/autoresearch` 技能进行系统性迭代，或开 agents team 进行多角度对抗性分析。困惑时停下来，指出不清楚之处，要求澄清再行动。
9. **强制最小可行方案优先（MVP-First）**：任何需求实施必须严格遵循以下流程：
   - **识别 MVP**：提炼出能独立交付价值、可独立验证的最小功能子集，严禁一次性铺开所有功能。
   - **MVP 必须包含详尽验收用例**：可执行的测试命令、预期输入/输出、边界条件，验收用例必须在编码前定义完毕。
   - **MVP 必须包含原子化 TodoList**：每个 item 可在 1-2 分钟内完成，标注前置依赖关系。
   - **非必要步骤列为可选项**：标注优先级（P1/P2/P3），仅在 MVP 验收通过且用户明确要求后才可实施。
   - 检验标准：资深工程师会觉得这过于复杂吗？如果是，简化。
10. **强制基于事实回答（禁止幻觉）**：所有回答必须基于实际文件内容和可验证的事实，严禁凭空捏造。先读后答，引用出处，标注代码片段、配置项、文件路径的来源文件及行号。不确定时明确声明"我无法确认此信息"，禁止编造不存在的文件路径、函数名、API。
11. **强制优先查阅 Midscene 源码**：本项目基于 [Midscene](https://github.com/web-infra-dev/midscene) 继续开发，所有 midscene 相关功能必须以官方仓库为准。
    - 涉及 Midscene 功能时，**必须优先**使用 `/documentation-lookup` 技能查阅官方仓库文档或源码
    - 禁止凭模型自身训练数据的记忆来实现 midscene 相关功能
    - 涉及 midscene 的 API、参数、行为时，必须引用官方文档 URL 或源码文件路径
    - 若官方文档无法解答，再尝试其他方式（WebSearch、读本地 node_modules 等）

## 编码原则（实践指南）

> 源自 Andrej Karpathy 关于 LLM 编码陷阱的观察：模型会代你做错误假设，不管理困惑，不呈现权衡，不提出异议，且倾向于过度复杂化。

### 原则 1：编码前思考

**不要假设。不要隐藏困惑。呈现权衡。**

- **明确说明假设** — 不确定时，询问而非猜测
- **呈现多种解释** — 存在歧义时，不要默默选择一种
- **适时提出异议** — 如果存在更简单的方案，说出来
- **困惑时停下来** — 指出不清楚之处，要求澄清再行动

### 原则 2：简洁优先

**用最少的代码解决问题。不要过度推测。**

- 不添加要求之外的功能
- 不为一次性代码创建抽象
- 不添加未要求的"灵活性"或"可配置性"
- 不为不可能发生的场景做错误处理
- 如果 200 行可写成 50 行，重写它
- 检验标准：资深工程师会觉得这过于复杂吗？如果是，简化。

### 原则 3：精准修改

**只碰必须碰的。只清理自己造成的混乱。**

编辑现有代码时：

- 不"改进"相邻的代码、注释或格式
- 不重构没坏的东西
- 匹配现有风格，即使你更倾向于不同写法
- 注意到无关的死代码时，提一下但不要删除

当改动产生孤儿代码时：

- 删除因你的改动而变得无用的导入/变量/函数
- 不删除预先存在的死代码，除非被要求
- 检验标准：每一行修改都应该能直接追溯到用户的请求。

### 原则 4：目标驱动执行

**定义成功标准。循环验证直到达成。**

将指令式任务转化为可验证的目标：

| 不要这样做... | 转化为... |
|--------------|-----------------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

多步骤任务须说明计划：

```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
```

### 权衡说明

这些原则倾向于**谨慎而非速度**。对于琐碎任务（简单的拼写修正、显而易见的一行修改），自行判断——并非每个改动都需要完整严谨流程。目标是减少非琐碎工作中的代价高昂的错误，而不是拖慢简单任务。

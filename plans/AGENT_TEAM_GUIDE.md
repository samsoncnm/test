# Agent Team 使用指南 & 计划模板

> 本目录存放复杂任务的讨论与计划文档。
> 命名格式：`YYYYMMDD-<简短标题>.md`

---

## 什么时候需要建立计划

**必须建立计划**（触发条件，满足任一即建档）：

- 任务涉及 3+ 文件修改
- 有外部 API / 数据库 / 第三方服务依赖
- 存在多个并行开发分支或 feature flag
- 历史上有类似任务失败或回滚的记录
- 单次修改预估超过 20 文件或 500 行 diff
- 任务预计超过 2 小时工作时间

**可选建立计划**：

- 用户明确要求先出计划再执行
- AI 自行判断风险不明确，需要对齐

## 多角色对抗性思考框架

### 三角色视角

每个计划必须回答以下三个角色的核心问题：

#### 角色 1：实现者 (Implementor)
> "这个能写出来吗？"

- [ ] 关键路径代码能否写出来？有没有未知技术难点？
- [ ] 边界条件和异常情况有哪些？
- [ ] 如何设计测试覆盖 happy path 和 sad path？
- [ ] 最少需要改几个文件？改动量是否在阈值内？
- [ ] 依赖是否都已就绪？版本是否兼容？

#### 角色 2：Reviewer (Critical Friend)
> "这段代码 3 个月后还看得懂吗？"

- [ ] 命名是否自解释？注释是否说明了 why 而非 what？
- [ ] 是否有隐式依赖（全局状态、共享变量、环境变量）？
- [ ] 是否引入了新的复杂度？能否用更简单的方案替代？
- [ ] 有没有潜在的 security 问题？
- [ ] 回滚成本高吗？是否可逆？

#### 角色 3：运维 / SRE (Production Guardian)
> "上线后会挂在哪里？"

- [ ] 部署顺序是什么？有没有先决条件？
- [ ] 如果失败了，如何回滚到上一个稳定版本？
- [ ] 有没有 metric / log / alert 验证功能正常？
- [ ] 流量高峰 / 网络抖动 / 服务重启时会发生什么？
- [ ] 数据迁移是否需要特殊处理？能否支持 zero-downtime？

### 对抗性假设挑战清单

每个计划必须对以下假设进行压测：

| # | 对抗性问题 | 如果假设不成立怎么办？ |
|---|-----------|---------------------|
| 1 | 外部 API / 服务是否一定可用？ | 是否有 fallback？超时多少秒？ |
| 2 | 数据格式是否一定符合预期？ | 是否有 schema 校验？格式变化如何感知？ |
| 3 | 用户量 / 请求量是否在预期范围内？ | 是否有 rate limit？性能基线是多少？ |
| 4 | 我读取的代码/文档是否是最新的？ | 是否与生产环境实际状态一致？ |
| 5 | 我的改动对其他模块是否有未知影响？ | 是否有 guard 命令回归测试？ |
| 6 | 权限和环境变量是否一定就绪？ | 是否需要新增配置项？谁来设置？ |
| 7 | 时间估算是否考虑了人的因素？ | 上下文切换、中途被打断怎么办？ |

## 使用流程

```
1. 触发条件满足 → 创建 plans/YYYYMMDD-<name>.md
2. 填写三角色视角压测 + 对抗性假设挑战
3. 若有不确定性 → 用 sequential-thinking 技能模拟辩论
4. 用户确认计划 → 开始执行
5. 执行中记录决策 → 追加到"决策记录"章节
6. 任务完成后 → 标记状态为"已完成"
```

## 常见反模式（避免清单）

- **没有假设清单就开工**：先射箭后画靶
- **只有乐观路径**：从不考虑"如果 XX 失败了怎么办"
- **计划写完不更新**：实际情况与计划脱节
- **三角色走过场**：每个角色问 3 个问题但问题没有实质内容
- **计划过于详细**：导致启动成本过高，反而不想建计划

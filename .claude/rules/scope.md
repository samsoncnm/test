# Scope 与工作模型

> 核心：控制变更边界，确保每次只做一件事。

## 原则 1：约束即赋能 (Constraint = Enabler)

- **Scope 先行**：任何任务开始前必须定义 in-scope 文件范围，out-of-scope 文件禁止修改
- **单一目标**：每次迭代聚焦一个可度量的成功标准，禁止同时追多目标
- **时间预算**：预估每轮迭代成本，高成本操作（大量文件生成、外部 API 调用）必须预先拆分
- **验收用例反推质量**：每个需求先定义验收用例，再编写代码；代码是验收用例的实现，而非验收用例服务于代码

## 原则 2：策略与战术分离 (Strategy vs. Tactics)

- **人类定方向**：等待用户明确 Goal 和 Scope，再执行战术层面的迭代
- **AI 主管控**：在既定方向内自主迭代，无需每步确认；遇到阻塞（缺权限、外部依赖不可达、需人工判断）立即上报

## 原则 3：任务拆解与注意力收敛 (Decompose & Focus)

> 上下文越长注意力越稀薄。必须主动拆解、主动收敛，不因上下文膨胀而放松标准。

- **强制 TodoList**：任何多步骤任务开始前必须生成 TodoList，每个 item 为原子操作，禁止模糊 item
- **单次变更边界**：每次修改只动一个逻辑点，改完验证再动下一个；禁止"一起改完再验证"
- **注意力警戒线**：对话上下文超过 200 行时，立即暂停、重读 CLAUDE.md、重新确认 in-scope，再继续
- **收敛检查点**：每个 sub-task 完成时自问"这个改动是否还在原始 scope 内？"，超界立即打住
- **禁止并行发散**：同时推进多个独立方向是注意力分散的根源，必须逐一完成、逐一验证

## 原则 4：命令执行必须有时限 (No Open-Ended Commands)

- **超时强制**：任何 shell 命令必须带 `--timeout` 或 `timeout` 参数
- **分片执行**：长时间操作（数据处理、编译、测试）必须预估时间并分片，禁止单次超过 10 分钟的无监控运行
- **超时上报**：命令超时后必须上报原因和当前状态，由用户决定是否继续，禁止自行重试超过 2 次

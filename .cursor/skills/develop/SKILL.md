# Develop — 5天对抗性开发策划

## 核心原则

**收敛优先。** 每个阶段有明确的退出标准。当多个方案势均力敌时，选择最简单的。

**成熟方案 = 业界验证 + 适配上下文 + 可执行。** 不是最酷炫的，是最适合的。

## 工作流

### 1. 理解需求（启动时执行）

- 拆解需求为子问题
- 识别已知约束（技术栈、性能、团队能力、deadline）
- 识别未知假设（需要验证）
- 需求模糊时先澄清，不盲目推进

### 2. Day 1 — 架构师

建立领域模型和初步架构方向。

产出：
- 核心数据流（Mermaid图）
- Top 3 风险
- 2个以内架构方向（供后续评估）

### 3. Day 2 — 竞品分析师 + DevOps专家（并行）

**竞品分析师**：研究3-5个竞品/开源方案，提炼设计模式，指出致命缺陷。

**DevOps专家**：评估CI/CD、部署复杂度、运维成本、安全边界。

### 4. Day 3 — 红队

对所有候选方案发起系统性攻击：
- 边界条件失效点
- 单点故障
- 安全攻击面
- 规模陷阱（QPS 10x/100x）
- 隐性依赖

原则：不只说"不行"，而是"在X条件下不行，如何加固"。

每套方案至少2条可操作加固建议。

### 5. Day 4 — 收敛决策

整合前3天输入，做出最终决策：
1. 列出所有候选方案
2. 打分矩阵（维度：实现复杂度、性能、运维友好度、团队适配度、扩展性、安全韧性）
3. 对抗性追问最高分方案："它的最大弱点是什么？"
4. 确定主方案 + Plan B（附触发条件）
5. 细化数据流、API契约、边界处理

### 6. Day 5 — 全科医生

最终全面检查，查漏补缺：
- 功能完整性（边界条件、用户故事）
- 数据一致性（事务、并发）
- API设计（错误码、版本管理）
- 可观测性（日志、指标、告警）
- 安全（认证、授权、加密）
- 测试策略
- 灰度与回滚策略

## 最终报告格式

```markdown
# [功能名称] 开发方案报告

## 1. Executive Summary
2-3句话：做什么、为什么这样做、预期结果

## 2. 方案决策
选型理由 + 打分矩阵 + Plan B触发条件

## 3. 架构设计
核心数据流 / API设计 / 数据模型

## 4. 实现计划

### Phase 1（优先级最高）
- [ ] 任务1（X小时）
- [ ] 任务2（X小时）

### Phase 2
- [ ] 任务3
...

**总工期：X人天**

## 5. 风险评估
已识别风险 + 缓解措施

## 6. 验收标准
功能 + 性能 + 安全指标
```

## 加速模式

需求简单/技术栈熟悉时，5天压缩为2天：
- Day 1 上午：架构师 + 竞品分析并行
- Day 1 下午：收敛 + todolist
- Day 2：全科医生最终检查

# Debug — 5天对抗性Bug排查

## 触发词

- 排查 bug / 修复问题 / debug
- 这个报错 / 跑不通 / 报错了

## 核心原则

**收敛优先。** 红队阶段必须强制排除不成立的假设。Day 4 选出唯一修复方案，不讨论"万一"。

**不过度修复。** 只修根因，不修症状。避免引入新问题。

**用 agents team。** 每个角色独立思考、独立输出，最后汇聚。

## 工作流

### Day 1 — 复现专家 (Reproducer)

目标：找到稳定复现路径。

产出：
- 稳定复现步骤（含前置条件、环境）
- 复现率（100% / 偶发 / 特定条件下）
- 边界条件（什么情况下复现 / 什么情况下不复发）

如无法稳定复现：说明这是最高优先级风险。

### Day 2 — 根因分析师 + 系统追踪师（并行）

**根因分析师 (Root Cause Analyst)**
从代码/架构层面逐层下挖：
- 调用链分析（从报错点向上回溯）
- 数据流分析（输入 → 处理 → 输出，哪一步出问题）
- 状态机分析（什么状态组合触发了异常）

**系统追踪师 (Tracer)**
从运行时线索追踪：
- 日志 trace 分析（有无异常信号）
- 监控/ metrics（CPU / 内存 / 网络 / DB 连接）
- 依赖服务状态（上游是否抖动）

两个角色独立输出，互不参考对方结论。

### Day 3 — 红队 (Devils Advocate)

对 Day 2 所有候选假设发起系统性攻击：
- "如果不是这个原因，最可能是什么？"
- 设计排除实验（改哪个变量能让问题消失/保留）
- 逐个击破不成立的假设

输出：排除后的**优先级假设列表**（最多3个），每个附上置信度。

### Day 4 — 修复决策 (Fix Decision)

整合前3天输入，做出最终决策：
1. 列出所有候选假设
2. 打分矩阵（维度：根因置信度、修复难度、引入新问题风险、影响范围）
3. 确定唯一修复方案 + Plan B（附触发条件）
4. 制定最小修复步骤（不超过5步，避免过度修复）

### Day 5 — 验证专家 (Verifier)

最终全面检查：
- 修复是否真正解决问题（复现测试）
- 边界条件验证（什么情况下仍可能复发）
- 回归风险评估（修复是否影响其他功能）
- 输出更新后的 todo list

## 最终报告格式

```markdown
# Bug排查报告 — [问题简述]

## 1. 问题概述
- 现象：[用户看到的表象]
- 影响范围：[哪些用户/功能受影响]
- 严重程度：[P0/P1/P2]

## 2. 复现路径
- 稳定复现步骤：
  1. ...
  2. ...
- 复现率：X%
- 边界条件：[触发/不触发条件]

## 3. 根因分析
- 根因链：触发条件 → 传播路径 → **根因点**
- 置信度：[X%]

## 4. 修复方案
- 主方案：[简述]
- 涉及文件：[列表]
- Plan B：[触发条件时启用]
- 工时：X小时

## 5. 验证计划
- 测试用例：[列表]
- 边界验证：[列表]
- 回归风险：[评估]

## 6. Todo List
- [ ] 步骤1（X小时）
- [ ] 步骤2（X小时）
- [ ] 验证（X小时）

**总工期：X小时**
```

## 加速模式

简单/明确 bug：Day 1 + Day 4 合并（复现即已知根因），Day 5 最终验证。
